# Advanced Algorithms Review Plan

## 1. 题目

- 课程：高级算法（Spring 2026）
- 论文：*A Hash Table Without Hash Functions, and How to Get the Most Out of Your Random Bits*
- 论文文件：[`paper/A Hash Table Without Hash Functions, and How to Get the Most Out of Your Random Bits.pdf`](paper/A%20Hash%20Table%20Without%20Hash%20Functions,%20and%20How%20to%20Get%20the%20Most%20Out%20of%20Your%20Random%20Bits.pdf)
- 选题说明：[`notes/00-论文选题.md`](notes/00-%E8%AE%BA%E6%96%87%E9%80%89%E9%A2%98.md)

本仓库用于记录本次 review 的阅读、理解、写作、修改与 AI 使用过程，以满足课程对“可在线查看、带版本历史的过程记录”的要求。

## 2. 作业目标

根据课程要求，本次 review 计划完成以下内容：

1. 解释论文试图解决什么问题，以及这个问题为什么重要。
2. 说明论文与课程中 Hashing、Bloom Filter、Perfect Hashing、Cuckoo Hashing 等内容的联系。
3. 系统梳理相关研究：
   - 论文正文中提到的前置工作；
   - 论文未展开但与主题密切相关的工作；
   - 论文发表后的后续研究。
4. 详细解释论文提出的核心技术、设计直觉与关键创新点。
5. 尽可能严谨地解释其实现方式、正确性证明思路以及性能优势来源。
6. 在最终 review 中明确写出参与者贡献与 AI 工具使用情况。

## 3. 选题理由

当前选择这篇论文，主要基于以下考虑：

1. 主题与课程内容高度相关，尤其适合从“哈希表是否一定依赖哈希函数”这一角度展开。
2. 问题本身有较强的概念冲击力，适合写成一篇以“问题意识 + 技术拆解”为核心的 review。
3. 可以自然联系随机化数据结构、随机比特使用效率、succinct dictionary 等更深层问题。
4. 相关工作链条比较清晰，便于做前后文献梳理。

更完整的选题思考见 [`notes/00-论文选题.md`](notes/00-%E8%AE%BA%E6%96%87%E9%80%89%E9%A2%98.md)。

### 4.2 review 预期结构

最终 review 暂定按以下结构推进：

1. 引言：问题背景与论文定位。
2. 问题定义：传统哈希表依赖什么，论文试图移除什么。
3. 相关工作：前置研究、平行研究与后续研究。
4. 核心思想：不用哈希函数的哈希表为何可能成立。
5. 技术细节：数据结构设计、随机性使用方式、操作流程。
6. 正确性与复杂度：主要证明思路与关键界。
7. 论文评价：创新点、适用范围、局限与启发。
8. 贡献说明与 AI 使用说明。

## 5. 过程安排

为满足“过程分 70%”的要求，本项目将把过程记录本身作为正式工作的一部分来维护。

### 阶段一：获取与初读论文

- 确认原文版本并完成通读。
- 记录论文中的问题定义、模型假设、主要定理与不理解之处。
- 形成第一版结构化笔记，例如 `notes/01-初读与问题整理.md`。

### 阶段二：补背景与相关工作

- 回看课程中与 hashing 相关的知识点。
- 补读论文引用的关键前置工作。
- 查找论文之后的相关研究，整理时间线与关系图，例如 `notes/02-相关工作梳理.md`。

### 阶段三：拆解核心技术

- 逐段理解论文的数据结构设计。
- 解释每个关键步骤背后的直觉。
- 把主要证明拆成可以被同学独立检查的逻辑链条，例如 `notes/03-核心技术与证明.md`。

### 阶段四：撰写初稿

- 先完成问题背景、相关工作与核心思想部分。
- 再补技术细节、证明思路与评价部分。
- 保留草稿痕迹，不做“一次性成稿”式提交，必要时继续新增 `notes/04-写作提纲.md` 一类文件。

### 阶段五：修订与定稿

- 补强表述不严谨或跳步过大的部分。
- 核对引用、术语、符号与结论是否一致。
- 明确写出成员贡献、修改过程与 AI 使用记录。

## 6. 仓库使用约定

为便于教师或助教检查版本历史，当前仓库按如下方式使用：

- `paper/`：保存论文原文。
- `paper/`：保存论文原文与双语阅读版 PDF。
- `notes/`：保存按编号递增组织的过程材料，如选题、阅读笔记、问题清单、资料整理等。
- `review/`：保存 review 的草稿与后续正式写作材料。
- `ai-log/`：保存按编号递增组织的 AI 辅助理解、检索、问答与整理记录。

后续每次重要推进，尽量体现在：

1. 新增或修改对应编号的 `notes/xx-xxxx.md`；
2. 更新 review 草稿；
3. 补充 AI 使用记录；
4. 通过 Git 历史保留修改轨迹。

## 8. 写作原则

这份 review 的写作希望尽量做到：

- 不只复述论文结论，而是把问题背景、技术动机和证明逻辑讲清楚。
- 不把“看起来对”当作“已经理解”，对关键定义、算法步骤和推导过程都要能展开解释。
- 尽量让没有读过原论文的同学也能顺着 review 独立理解并检查主要论证。
- 过程记录与正文同步推进，避免最后一次性整理。
